# 后台管理系统接口设计方案

> 版本：v4 · 更新日期：2026-05-29
> 关联文档：[模特资料与当日通告接单-H5设计方案.md](./模特资料与当日通告接单-H5设计方案.md)（v3）、[Figma高保真设计方案.md](./Figma高保真设计方案.md)、[部署架构.md](./部署架构.md)
> 适用范围：后台管理系统（工作室自助上下架）+ 公开 H5 接口
> 部署形态：**单台香港 VPS 自部署 + Cloudflare DNS proxied · 不使用 Docker · 海外部署 · 不备案 · 不接审核**（v4 推翻 v3 的 Cloudflare 全栈 Serverless，详见 §十 业主决策 2026-05-29）

## 〇、本方案是什么

本文档定义后台管理系统的**接口契约与数据模型**，不含具体代码实现。回答以下问题：

- 系统由哪些角色 / 数据实体构成
- API 长什么样（路径、入参、出参、错误码）
- 鉴权怎么做、权限怎么划分
- 安全与运维的硬性约束

按本方案产出后，前端 / 后台 / 移动端可并行开发，依据接口契约解耦。

## 一、系统架构

### 1.1 角色

| 角色 | 说明 | 权限粒度 |
| --- | --- | --- |
| **Owner** | 工作室主理人 | 全部 + 账号管理 |
| **Admin** | 经纪人 | 模特 / 名单 / 媒体 / 档期 CRUD |
| **Operator** | 运营助理（可选） | 仅当日名单 + 媒体上传，不能改模特资料 |

### 1.2 总体架构

```
                          Cloudflare DNS（proxied 模式）
                                       │
                          [DDoS / WAF / CDN / 隐藏源 IP]
                                       │
                                       ▼
                ┌──────────────────────────────────────────┐
                │       香港 VPS（2C / 2G / 40G）            │
                │                                          │
                │   Caddy 2（Cloudflare Origin CA 证书）     │
                │   ├── chiyan.com           → H5 静态文件   │
                │   ├── admin.chiyan.com     → Admin 静态    │
                │   ├── api.chiyan.com       → reverse_proxy │
                │   └── /media/*             → 本地磁盘静态   │
                │                                          │
                │   ┌────────────────────────────────────┐ │
                │   │ chiyan-api.service (systemd)        │ │
                │   │   Node.js 22 + Hono on Node         │ │
                │   │   :3000 localhost                   │ │
                │   └────┬──────────┬────────────┬───────┘ │
                │        │          │            │         │
                │  postgresql.service  redis.service        │
                │   (apt 原生)         (apt 原生)           │
                │        │          │            │         │
                │        ▼          ▼            ▼         │
                │   PostgreSQL 16  Redis 7   /var/chiyan/  │
                │   :5432 local    :6379 local   media/    │
                │                                          │
                └──────────────────────────────────────────┘
                                       │
                          ┌────────────┴────────────┐
                          ▼                          ▼
                    Backblaze B2                  Sentry
                  （pg_dump + media               （错误监控）
                    rsync 每日备份）
```

> 单机部署，所有进程用 systemd 托管，**不使用 Docker**。详细 unit 配置、Caddyfile、内存预算见 [部署架构.md](./部署架构.md)。

### 1.3 技术选型（2026-05-29 第二轮锁定）

| 维度 | 选型 | 备注 / 约束 |
| --- | --- | --- |
| 运行时 | **Node.js 22 LTS** | VPS 上 systemd 托管，单进程；不使用 Docker |
| 后端框架 | **Hono on Node**（`@hono/node-server` adapter） | 保留 Hono 的轻量路由 + 中间件能力；可自由用 Node 原生 `fs / net / child_process` |
| ORM | **Drizzle ORM** | driver 由 v3 的 `drizzle-orm/neon-http` 切换为 `drizzle-orm/node-postgres` |
| 数据库 | **PostgreSQL 16**（VPS apt 原生安装） | 走 `pg` Pool TCP 长连，仅监听 `127.0.0.1`；无 scale-to-zero，但 2G 机器够用 |
| 对象存储 | **VPS 本地磁盘** `/var/chiyan/media` | 不再使用 R2 / OSS / 七牛；目录结构 `models/<id>/{cover,gallery,thumbs,originals}` |
| 静态托管 / 反代 / TLS | **Caddy 2** | Cloudflare Origin CA 证书（15 年）；同时托管 H5、Admin 静态文件，并把 `/media/*` 直接发本地磁盘 |
| 图片处理 | **sharp**（Node native） | 上传时一次性生成水印图 + 缩略图 + WebP，落盘静态目录；原图放 `originals/` 不走 CDN |
| 缓存 / 限流 / jti 黑名单 | **Redis 7**（VPS apt 原生安装） | TCP 长连,`ioredis` 客户端；仅监听 `127.0.0.1` |
| 前端框架 | Vite + React 18 + TypeScript + TailwindCSS + shadcn/ui | H5 与 Admin 同一 monorepo，分别 build 后由 Caddy 托管 |
| 前端部署 | **Caddy 静态文件托管**（不再用 Cloudflare Pages） | `pnpm build` 产物 rsync 到 VPS 静态目录，Caddy 自带 gzip / brotli |
| 后台认证 | **自建 JWT + HttpOnly Cookie + 强制 TOTP + jti 黑名单**（不引入 Lucia） | jose 签名 ES256；jti 黑名单存 Redis |
| 密码哈希 | **bcrypt**（native，cost 12） | 替换 v3 的 bcryptjs，native 实现快约 10×，2C 机器一次登录 ~80ms |
| 加密 | **node:crypto** AES-256-GCM | 替换 v3 的 WebCrypto；密钥从环境变量读取，分版本号 |
| TOTP | otpauth | 不变，纯 JS |
| 短信 / 邮件通道 | ❌ **不接入** | 业主 2026-05-28 决策：零 PII，账号邀请与密码重置由 Owner 后台生成一次性密码、线下交付 |
| DNS / CDN | **Cloudflare（proxied 模式）** | 隐藏源 IP + DDoS + WAF；大陆访问 best-effort，必要时叠加 China-friendly CDN |
| CI / 部署 | **GitHub Actions → SSH 到 VPS** | `git pull && pnpm install --frozen-lockfile && pnpm -r build && systemctl restart chiyan-api` |
| 监控 | **Sentry + Caddy access log + systemd journal** | 不依赖 Cloudflare Analytics |
| 备份 | **`pg_dump` + `rsync media/` → Backblaze B2** | cron 每日全量；保留 30 天 |

> 月成本：VPS 已购固定 + Backblaze B2 几美元 + Cloudflare 免费层 + Sentry 免费层。不备案、不接入内容审核（业主 2026-05-28）。

### 1.4 资源预算（2G RAM 是硬约束）

| 进程 | 预算 | 关键调优 |
| --- | --- | --- |
| PostgreSQL 16 | ~512 MB | `shared_buffers=256MB`、`effective_cache_size=1GB`、`work_mem=8MB`、`max_connections=20` |
| Redis 7 | ~128 MB | `maxmemory 128mb`、`maxmemory-policy allkeys-lru`、关闭 RDB 持久化保留 AOF |
| Node API（Hono）| ~256 MB | 单进程 `--max-old-space-size=384`，无 cluster |
| Caddy 2 | ~64 MB | 默认即可 |
| sharp 临时峰值 | ~256 MB | 限制并发：图片处理走 `p-limit(2)`，避免大图同时解码 OOM |
| 系统 + buffer | ~512 MB | 留足 |
| **合计目标** | **≤ 1.7 GB** | 留 ~300 MB 给突发请求与 page cache |

详细操作参数与 systemd `MemoryMax` 配置见 [部署架构.md](./部署架构.md)。

## 三、认证与鉴权

### 3.1 登录方式

**唯一登录方式：用户名 + 密码 + TOTP**（业主 2026-05-28 决策，零 PII）

| 步骤 | 输入 | 行为 |
| --- | --- | --- |
| 1 | username + password | 验证账密，返回 `requires_totp: true` + 临时 challenge token（TTL 5 min） |
| 2 | challenge + TOTP code | 验证 TOTP，返回正式 Access Token + Refresh Cookie |

**首次登录强制流程**：

- `must_change_password = true` → 登录后强制弹出改密页面，未改密前所有非 `/auth/*` 接口返回 40301
- `totp_enrolled = false` → 改密后强制 TOTP 绑定流程（展示 QR + 输入一次验证码确认）
- 完成上述两步后写入 `must_change_password = false` + `totp_enrolled = true`，正式放行

**所有角色强制 TOTP**：Owner / Admin / Operator 均不可关闭。

### 3.1.1 密码策略

| 项 | 要求 |
| --- | --- |
| 最短长度 | 12 字符 |
| 复杂度 | 至少包含 3 类：大写字母 / 小写字母 / 数字 / 符号 |
| 哈希算法 | bcrypt cost 12 |
| 一次性密码（Owner 创建 / 重置时生成） | 20 字符 URL-safe random，明文仅响应一次后立即落 hash |
| 密码 history | 不允许复用最近 5 次密码 |
| 失败锁定 | 连续 5 次失败 → 账号锁定 15 分钟（`locked_until`） |

### 3.2 Token 机制

- **Access Token**：JWT，TTL 2 小时，放 Authorization Header
- **Refresh Token**：HttpOnly + Secure + SameSite=Lax Cookie，TTL 14 天
- **退出**：服务端将 jti 加入 Redis 黑名单直到自然过期

### 3.3 权限矩阵

| 资源 | Owner | Admin | Operator |
| --- | --- | --- | --- |
| 模特 CRUD | ✅ | ✅ | 只读 |
| 当日名单 | ✅ | ✅ | ✅ |
| 媒体上传 | ✅ | ✅ | ✅ |
| 媒体删除 | ✅ | ✅ | ❌ |
| 档期管理 | ✅ | ✅ | 只读 |
| 账号管理 | ✅ | ❌ | ❌ |
| 审计日志 | ✅ 全部 | 只看自己 | 只看自己 |

## 四、API 接口清单

### 4.1 通用约定

- Base URL：`/api/v1/`
- 路由：kebab-case
- 请求 / 响应：JSON
- 时间：ISO 8601 UTC（`2026-05-25T14:30:00Z`）
- 日期：`YYYY-MM-DD`

**统一响应格式**

```json
{
  "code": 0,
  "message": "ok",
  "data": { },
  "request_id": "req_a1b2c3",
  "timestamp": 1716620000
}
```

错误：
```json
{
  "code": 40401,
  "message": "模特不存在",
  "request_id": "req_a1b2c3"
}
```

### 4.2 认证

| Method | Path | 鉴权 | 说明 |
| --- | --- | --- | --- |
| POST | `/auth/login` | 无 | 第一步：username + password，返回 challenge token |
| POST | `/auth/login/totp` | 凭 challenge | 第二步：challenge + TOTP code，返回 Access Token + Refresh Cookie |
| POST | `/auth/refresh` | Refresh Cookie | 刷新 Access Token |
| POST | `/auth/logout` | 鉴权 | 退出（jti 拉黑） |
| GET | `/auth/me` | 鉴权 | 当前登录信息（含 `must_change_password` / `totp_enrolled` 状态） |
| POST | `/auth/change-password` | 鉴权 | 修改自己密码（旧密码 + 新密码） |
| POST | `/auth/totp/setup` | 鉴权 + `totp_enrolled=false` | 返回 secret + otpauth URL（前端生成 QR） |
| POST | `/auth/totp/verify` | 鉴权 + `totp_enrolled=false` | 提交一次 TOTP code 完成绑定 |

## 五、规范细节

### 5.1 分页

```
?page=1&page_size=20&order_by=created_at&order=desc
```

返回：
```json
{ "items": [...], "total": 145, "page": 1, "page_size": 20 }
```

### 5.2 筛选

模特列表：`?type=平面&style=御姐&q=林夕&status=active`

业主决策：模特规模小，无需 ES，PostgreSQL `ILIKE` + `style_tags @> '["御姐"]'` 即可。

### 5.3 错误码

| Code | 含义 |
| --- | --- |
| 0 | 成功 |
| 40001 | 参数错误 |
| 40101 | 未登录 |
| 40301 | 无权限 |
| 40401 | 资源不存在 |
| 40901 | 冲突（重复编号、并发更新） |
| 42901 | 限流 |
| 41001 | 资源已下架（Gone） |
| 50001 | 服务器错误 |
| 50301 | 上游不可用（PostgreSQL / Redis / 本地磁盘 / Cloudflare API） |

业务子码可在 `data.sub_code` 里追加。

### 5.4 幂等

写接口（POST / PUT / PATCH）支持 `Idempotency-Key` Header，服务端缓存 24 小时结果。

## 六、关键场景流程

### 6.1 新增模特 + 上传图片

```
1. 后台前端 → POST /admin/media/upload (multipart)   直接上传到 Node API
2. Node API：
   - 校验 MIME + magic number + 文件大小
   - 算 sha256，查重
   - sharp 同步生成：水印图 / 缩略图 / WebP / 原图分别落 /var/chiyan/media/...
   - 写 MediaAsset 记录
   - 返回 { asset_id, cdn_url: "/media/models/<id>/cover.webp" }
3. 前端     → POST /admin/models                     提交模特信息 + cover_asset_id
4. 后端     → Cloudflare cache purge `/public/models`   刷新公开缓存
5. 返回 model.code
```

> v3 的 sign → 客户端直传 R2 → register 三步流程在 v4 合并为单次 multipart POST，sharp 处理在请求内同步完成（限 `p-limit(2)` 并发，避免 2G 机器 OOM）。前端只需要处理一个上传进度条。

### 6.2 维护当日名单（最高频操作）

```
1. GET /admin/models?status=active      拉候选
2. 拖拽到当日名单
3. PUT /admin/roster
   body: { date, model_ids, note }
4. 后端：
   - 写入 / 覆盖 DailyRoster
   - Cloudflare cache purge `/public/today?date=...`
   - 写审计日志
5. H5 下次访问即看到新数据（5 min CDN 缓存）
```

### 6.3 模特下架（隐私申请优先）

```
1. PATCH /admin/models/:id  { status: "archived" }
2. 后端：
   - 软删模特
   - 从所有 DailyRoster 移除其 ID
   - Cloudflare cache purge `/public/models/:code` + `/public/models` + `/public/today`
   - 审计日志记录原因
3. /public/models/:code 后续返回 41001 Gone
```

## 七、安全

### 7.1 限流（基于 Redis 滑动窗口）

| 接口类 | 限制 |
| --- | --- |
| 公开 API | 60 req/min/IP |
| 后台 API | 120 req/min/账号 |
| `/auth/login`（账密） | 10 req/h/username + 20 req/min/IP；连续 5 次失败账号锁定 15 min（见 §3.1.1） |
| `/auth/login/totp` | 10 req/5min/challenge token；challenge 用过即焚 |
| `/auth/change-password` / `/auth/totp/*` | 10 req/h/账号 |

超限返回 `429` + `Retry-After`；账号锁定返回 `40301` + `data.locked_until`。

### 7.2 文件上传

- MIME + magic number 双重校验
- 白名单：`jpg / jpeg / png / webp / mp4`
- 单文件 ≤ 20MB（图）/ 50MB（视频）
- **不接入内容审核**（业主决策 2026-05-28：邀请制 + 海外部署，不公开于中国互联网）
- **处理流水**：Hono multipart 接收 → magic number 校验 → sharp 同步生成多版本（水印图 / 缩略图 200w / 800w / WebP / AVIF）→ 落盘到 `/var/chiyan/media/models/<id>/{thumbs,gallery,originals}/`
- **并发约束**：图片处理走 `p-limit(2)`，单次 sharp 实例处理结束立即释放，避免 2G 机器同时解码大图 OOM
- **原图隔离**：`originals/` 目录仅 `chiyan-api.service` 可读（systemd `ReadWritePaths`），Caddy 不挂载；后台拉原图走 `/admin/media/:id/raw` 带 Token
- **水印副本**：sharp `composite()` 叠加半透明 PNG / SVG 水印，输出到 `gallery/` 与 `cover/`，挂 CDN
- **目录布局**：
  ```
  /var/chiyan/media/
    ├─ models/<model_id>/
    │   ├─ cover.webp / cover.avif
    │   ├─ gallery/  （水印副本，公开）
    │   ├─ thumbs/   （200w / 400w / 800w，公开）
    │   └─ originals/（无水印原图，仅后台带 Token）
    └─ studio/       （工作室通用素材）
  ```

### 7.3 敏感字段加密

- `real_name_enc`：`node:crypto` AES-256-GCM，密钥从 systemd `EnvironmentFile` 读，不写入 `.env` 文件直接进仓库
- `password_hash`：bcrypt native cost 12（不可逆）
- `totp_secret_enc`：AES-256-GCM
- **一次性密码明文**：仅在 Owner 创建账号 / 重置密码的响应里出现一次，不进日志、不进 Sentry、不写入 `audit_log.payload`
- **密钥轮换**：AES 密钥分版本号（`v1` / `v2`），加密时记录版本，解密自动按版本查表；轮换时先双写再下线旧版

### 7.4 CSRF / XSS

- 写接口要求 `X-CSRF-Token` Header（Cookie + Header 双验证）
- 输入 HTML 转义，富文本走白名单 sanitize
- Response Header：`Content-Security-Policy`、`X-Frame-Options: DENY`、`Strict-Transport-Security`

### 7.5 防截屏 / 防盗图

服务端能做的有限，**客户端水印是主防线**：
- 所有公开图片 CDN URL 带水印副本
- 原图不出 CDN，仅后台带 Token 访问
- 高敏感模特可标记 `watermark_level=high`，水印更明显

## 八、部署与运维

> 本节给出与接口契约相关的运维结论。具体 systemd unit、Caddyfile、备份脚本、故障预案见 [部署架构.md](./部署架构.md)。

### 8.1 环境

| 环境 | 用途 | 数据 | 运行位置 |
| --- | --- | --- | --- |
| dev | 开发联调 | mock | 开发机本地 |
| staging | 验收 | 脱敏镜像 | VPS 同机 `:3001` + 独立数据库 `chiyan_staging` |
| prod | 生产 | 真实 | VPS `:3000` + `chiyan_prod` |

> staging 与 prod 共用同一台 VPS，靠端口 + 数据库名隔离；正式上线前评估升级到双机方案。

### 8.2 部署架构（VPS 单机）

```
[Cloudflare DNS proxied + WAF + DDoS + 隐藏源 IP]
        │
        ▼
[香港 VPS 2C/2G/40G]
        │
        Caddy 2（443 / 80）── Cloudflare Origin CA 证书（15 年）
        ├── chiyan.com           → H5 静态：/var/www/chiyan-h5/dist
        ├── admin.chiyan.com     → Admin 静态：/var/www/chiyan-admin/dist
        ├── api.chiyan.com       → reverse_proxy 127.0.0.1:3000
        └── /media/*             → file_server /var/chiyan/media（仅水印 + 缩略 + 公开层）
        │
        chiyan-api.service（systemd / Node 22 / Hono / :3000 / localhost-only）
        │     │
        │     ├── pg Pool ─► postgresql.service（:5432 localhost）
        │     ├── ioredis ─► redis.service（:6379 localhost）
        │     └── fs ─► /var/chiyan/media（owner: chiyan-api，仅 originals/ 后台 Token 可读）
        │
        ↓（cron 每日 02:30）
        backup.sh ─► Backblaze B2：pg_dump_<date>.sql.gz + media-<date>.tar
```

部署单位：单台 VPS + 4 个 systemd unit（`postgresql.service` / `redis.service` / `caddy.service` / `chiyan-api.service`）。前端两个 SPA 是纯静态文件，不需要独立进程。

### 8.3 监控

- 接口耗时 P95 < 300ms（VPS 本机内网测得 + Caddy access log 二次验证），P99 < 800ms
- 错误率 > 1% 告警（Sentry）
- 资源告警：CPU > 80% / 内存 > 1.7G / 磁盘 > 30G / Postgres 慢查 > 500ms → systemd timer 触发邮件 + Sentry 自定义事件
- 健康检查：`api.chiyan.com/health`（数据库 + Redis 探活，10s 超时）；Caddy 上游探测 + 自动 retry
- 日志收集：systemd journal（结构化 JSON）+ Caddy access log 转 JSON；不接 ELK，保留 14 天

### 8.4 备份

- **DB**：`pg_dump -Fc chiyan_prod | gzip` 每日 02:30 写本地 → 上传 Backblaze B2，保留 30 天；每周日额外做一次 schema-only dump 留档
- **媒体**：增量 `rsync` 到 Backblaze B2 `b2://chiyan-media-mirror/`，每日 03:00
- **加密密钥**：`/etc/chiyan/secrets.env` 由 1Password 主理人保管，VPS 损毁场景下从 1Password 恢复
- **恢复演练**：每季度一次 staging 全量恢复，跑 smoke test，验证 RPO ≤ 24h、RTO ≤ 2h
- **审计日志**：1 年内可查，超期 dump 到 B2 冷存储

| # | 决策项 | 结论 | 影响章节 |
| --- | --- | --- | --- |

| 4 | 数据库 | ✅ **PostgreSQL 16**（VPS apt 原生），替换 v3 的 Neon Serverless；Drizzle driver 切换为 `node-postgres` | §一.3 |
| 5 | 缓存 | ✅ **Redis 7**（VPS apt 原生）+ ioredis TCP，替换 v3 的 Upstash REST | §一.3 |
| 6 | 对象存储 | ✅ **VPS 本地磁盘** `/var/chiyan/media` + Caddy 静态托管，替换 v3 的 Cloudflare R2 + presigned URL | §一.3、§四.5、§六.1、§七.2、§九.3 |
| 7 | 图片处理 | ✅ **sharp**（Node native）上传时同步生成水印 + 缩略 + WebP，替换 v3 的 Cloudflare `@cf/image-resizing` | §七.2 |
| 8 | 反代 / TLS | ✅ **Caddy 2** + Cloudflare Origin CA 证书（15 年），替换 v3 的 Pages + Workers 自动 TLS | §一.2、§八.2 |
| 9 | CI / 部署 | ✅ **GitHub Actions → SSH 到 VPS** 跑 `git pull && pnpm build && systemctl restart`，替换 v3 的 wrangler deploy | §一.3 |
| 10 | 密码 / 加密 | ✅ **bcrypt native cost 12** + **node:crypto AES-256-GCM**，替换 v3 的 bcryptjs + WebCrypto | §一.3、§七.3 |
| 11 | 后台认证 | ✅ **自建 JWT + jose ES256 + jti Redis 黑名单**，不引入 Lucia | §一.3 |
| 12 | 备份 | ✅ `pg_dump` + `rsync media/` 推 **Backblaze B2**，每日 cron；替换 v3 的 Neon PITR + R2 versioning | §八.4 |
| 13 | 资源预算 | ⚠️ **2G RAM 是硬约束**：Postgres 512M + Redis 128M + Node 256M + sharp 峰值 256M + Caddy 64M + 系统 512M ≈ 1.7G | §一.4 |

**决策动机**：业主 2026-05-29 明确表态："这个 H5 网站架构是在太麻烦了"。CF Workers 方案的工程税太高——
- Workers V8 Isolate 禁 `fs / net / child_process / TCP 长连`，强迫 `@neondatabase/serverless` HTTP 驱动、Upstash HTTPS REST、bcryptjs（CPU 慢 10×）、WebCrypto；每个选型都被运行时框死。
- 部署单元 3 份（Pages × 2 + Workers × 1），外部依赖 5 家（CF + Neon + Upstash + Sentry + R2），上线前任一 token / 配额都是硬阻塞。
- 图片水印依赖未验过的 `@cf/image-resizing`，水印效果有风险。

VPS 方案：一台机器装下全部，标准 Node 生态可用，sharp 水印一次跑通，备份就是 `pg_dump + rsync`。月成本固定（VPS 已购）+ Backblaze B2 几美元，长期更省。
