## 〇、本方案是什么

本文档定义后台管理系统的**接口契约与数据模型**，不含具体代码实现。回答以下问题：

- 系统由哪些角色 / 数据实体构成
- API 长什么样（路径、入参、出参、错误码）
- 鉴权怎么做、权限怎么划分
- 安全与运维的硬性约束

按本方案产出后，前端 / 后台 / 移动端可并行开发，依据接口契约解耦。

## 一、系统架构

### 1.1 角色

| 角色 | 说明 | 权限粒度 |
| --- | --- | --- |
| **Owner** | 工作室主理人 | 全部 + 账号管理 |
| **Admin** | 经纪人 | 模特 / 名单 / 媒体 / 档期 CRUD |
| **Operator** | 运营助理（可选） | 仅当日名单 + 媒体上传，不能改模特资料 |

### 1.2 总体架构

```
                       Cloudflare DNS
                              │
        ┌─────────────────────┼─────────────────────────┐
        │                     │                         │
H5 客户端                Admin Console             公开 API + 鉴权 API
（邀请制 URL 混淆）       （admin.* 子域）          （Cloudflare Workers + Hono）
        │                     │                         │
        └────► Cloudflare Pages ◄─────┘                 │
                                                        ├─► Neon Postgres（Serverless）
                                                        ├─► Upstash Redis（限流 + Session）
                                                        └─► Cloudflare R2（图片 / 视频）
```

### 1.3 技术选型（2026-05-28 已锁定）

| 维度 | 选型 | 备注 / 约束 |
| --- | --- | --- |
| 后端框架 | **Hono + TypeScript on Cloudflare Workers** | V8 Isolate，冷启动 5–10ms；不能用 Node 原生 `fs / net / child_process`；NestJS 装饰器 / Module 风格不适用 |
| ORM | **Drizzle ORM** | TS 类型完整；与 Workers 友好 |
| 数据库 | **Neon**（Serverless Postgres 15+） | 用 `@neondatabase/serverless` HTTP 驱动，不用 `pg` TCP 驱动；scale-to-zero |
| 对象存储 | **Cloudflare R2** | S3 兼容；**egress 免费**；presigned URL 直传 |
| 图片处理 | Cloudflare Images / Workers `@cf/image-resizing` | 按需缩放 / WebP / 水印一并解决 |
| 缓存 / 限流 / Session | **Upstash Redis**（HTTPS API） | Workers 不能持 TCP 连接，必须走 HTTP |
| 前端框架 | Vite + React 18 + TypeScript + TailwindCSS + shadcn/ui | H5 与 Admin 同一 monorepo，不同子域 |
| 前端部署 | **Cloudflare Pages** | 自动 PR 预览；与 Workers 同域无 CORS |
| 后台认证 | **Lucia Auth** + HttpOnly Cookie + TOTP | 用户名 + 密码 + TOTP，所有角色强制 TOTP；不存手机号 / 邮箱 |
| 短信 / 邮件通道 | ❌ **不接入** | 业主 2026-05-28 决策：零 PII，账号邀请与密码重置由 Owner 后台生成一次性密码、线下交付 |
| DNS / CDN | Cloudflare | 大陆访问 best-effort，必要时叠加 China-friendly CDN（CDN77 / BytePlus / EdgeNext） |
| CI | GitHub Actions → Wrangler deploy | — |
| 监控 | Cloudflare Analytics + Sentry | — |

> 全部按量计费，月活几百到几千的体量下月成本接近 $0。不备案、不接入内容审核（业主决策 2026-05-28）。

## 三、认证与鉴权

### 3.1 登录方式

**唯一登录方式：用户名 + 密码 + TOTP**（业主 2026-05-28 决策，零 PII）

| 步骤 | 输入 | 行为 |
| --- | --- | --- |
| 1 | username + password | 验证账密，返回 `requires_totp: true` + 临时 challenge token（TTL 5 min） |
| 2 | challenge + TOTP code | 验证 TOTP，返回正式 Access Token + Refresh Cookie |

**首次登录强制流程**：

- `must_change_password = true` → 登录后强制弹出改密页面，未改密前所有非 `/auth/*` 接口返回 40301
- `totp_enrolled = false` → 改密后强制 TOTP 绑定流程（展示 QR + 输入一次验证码确认）
- 完成上述两步后写入 `must_change_password = false` + `totp_enrolled = true`，正式放行

**所有角色强制 TOTP**：Owner / Admin / Operator 均不可关闭。

### 3.1.1 密码策略

| 项 | 要求 |
| --- | --- |
| 最短长度 | 12 字符 |
| 复杂度 | 至少包含 3 类：大写字母 / 小写字母 / 数字 / 符号 |
| 哈希算法 | bcrypt cost 12 |
| 一次性密码（Owner 创建 / 重置时生成） | 20 字符 URL-safe random，明文仅响应一次后立即落 hash |
| 密码 history | 不允许复用最近 5 次密码 |
| 失败锁定 | 连续 5 次失败 → 账号锁定 15 分钟（`locked_until`） |

### 3.2 Token 机制

- **Access Token**：JWT，TTL 2 小时，放 Authorization Header
- **Refresh Token**：HttpOnly + Secure + SameSite=Lax Cookie，TTL 14 天
- **退出**：服务端将 jti 加入 Redis 黑名单直到自然过期

### 3.3 权限矩阵

| 资源 | Owner | Admin | Operator |
| --- | --- | --- | --- |
| 模特 CRUD | ✅ | ✅ | 只读 |
| 当日名单 | ✅ | ✅ | ✅ |
| 媒体上传 | ✅ | ✅ | ✅ |
| 媒体删除 | ✅ | ✅ | ❌ |
| 档期管理 | ✅ | ✅ | 只读 |
| 账号管理 | ✅ | ❌ | ❌ |
| 审计日志 | ✅ 全部 | 只看自己 | 只看自己 |

## 四、API 接口清单

### 4.1 通用约定

- Base URL：`/api/v1/`
- 路由：kebab-case
- 请求 / 响应：JSON
- 时间：ISO 8601 UTC（`2026-05-25T14:30:00Z`）
- 日期：`YYYY-MM-DD`

**统一响应格式**

```json
{
  "code": 0,
  "message": "ok",
  "data": { },
  "request_id": "req_a1b2c3",
  "timestamp": 1716620000
}
```

错误：
```json
{
  "code": 40401,
  "message": "模特不存在",
  "request_id": "req_a1b2c3"
}
```

### 4.2 认证

| Method | Path | 鉴权 | 说明 |
| --- | --- | --- | --- |
| POST | `/auth/login` | 无 | 第一步：username + password，返回 challenge token |
| POST | `/auth/login/totp` | 凭 challenge | 第二步：challenge + TOTP code，返回 Access Token + Refresh Cookie |
| POST | `/auth/refresh` | Refresh Cookie | 刷新 Access Token |
| POST | `/auth/logout` | 鉴权 | 退出（jti 拉黑） |
| GET | `/auth/me` | 鉴权 | 当前登录信息（含 `must_change_password` / `totp_enrolled` 状态） |
| POST | `/auth/change-password` | 鉴权 | 修改自己密码（旧密码 + 新密码） |
| POST | `/auth/totp/setup` | 鉴权 + `totp_enrolled=false` | 返回 secret + otpauth URL（前端生成 QR） |
| POST | `/auth/totp/verify` | 鉴权 + `totp_enrolled=false` | 提交一次 TOTP code 完成绑定 |

## 五、规范细节

### 5.1 分页

```
?page=1&page_size=20&order_by=created_at&order=desc
```

返回：
```json
{ "items": [...], "total": 145, "page": 1, "page_size": 20 }
```

### 5.2 筛选

模特列表：`?type=平面&style=御姐&q=林夕&status=active`

业主决策：模特规模小，无需 ES，PostgreSQL `ILIKE` + `style_tags @> '["御姐"]'` 即可。

### 5.3 错误码

| Code | 含义 |
| --- | --- |
| 0 | 成功 |
| 40001 | 参数错误 |
| 40101 | 未登录 |
| 40301 | 无权限 |
| 40401 | 资源不存在 |
| 40901 | 冲突（重复编号、并发更新） |
| 42901 | 限流 |
| 41001 | 资源已下架（Gone） |
| 50001 | 服务器错误 |
| 50301 | 上游不可用（R2 / Neon / Upstash / 短信） |

业务子码可在 `data.sub_code` 里追加。

### 5.4 幂等

写接口（POST / PUT / PATCH）支持 `Idempotency-Key` Header，服务端缓存 24 小时结果。

## 六、关键场景流程

### 6.1 新增模特 + 上传图片

```
1. 后台前端 → POST /admin/media/sign            拿 R2 presigned PUT URL
2. 前端     → PUT  <r2-url>                     直传图片
3. 前端     → POST /admin/media/register        注册 asset，Workers 异步生成水印 + 缩略
4. 前端     → POST /admin/models                提交模特信息 + cover_asset_id
5. 后端     → Cloudflare cache purge `/public/models`   刷新公开缓存
6. 返回 model.code
```

### 6.2 维护当日名单（最高频操作）

```
1. GET /admin/models?status=active      拉候选
2. 拖拽到当日名单
3. PUT /admin/roster
   body: { date, model_ids, note }
4. 后端：
   - 写入 / 覆盖 DailyRoster
   - Cloudflare cache purge `/public/today?date=...`
   - 写审计日志
5. H5 下次访问即看到新数据（5 min CDN 缓存）
```

### 6.3 模特下架（隐私申请优先）

```
1. PATCH /admin/models/:id  { status: "archived" }
2. 后端：
   - 软删模特
   - 从所有 DailyRoster 移除其 ID
   - Cloudflare cache purge `/public/models/:code` + `/public/models` + `/public/today`
   - 审计日志记录原因
3. /public/models/:code 后续返回 41001 Gone
```

## 七、安全

### 7.1 限流（基于 Redis 滑动窗口）

| 接口类 | 限制 |
| --- | --- |
| 公开 API | 60 req/min/IP |
| 后台 API | 120 req/min/账号 |
| `/auth/login`（账密） | 10 req/h/username + 20 req/min/IP；连续 5 次失败账号锁定 15 min（见 §3.1.1） |
| `/auth/login/totp` | 10 req/5min/challenge token；challenge 用过即焚 |
| `/auth/change-password` / `/auth/totp/*` | 10 req/h/账号 |

超限返回 `429` + `Retry-After`；账号锁定返回 `40301` + `data.locked_until`。

### 7.2 文件上传

- MIME + magic number 双重校验
- 白名单：`jpg / jpeg / png / webp / mp4`
- 单文件 ≤ 20MB（图）/ 50MB（视频）
- **不接入内容审核**（业主决策 2026-05-28：邀请制 + 海外部署，不公开于中国互联网）
- 水印：Workers 用 `@cf/image-resizing` + 叠加 SVG / PNG 水印生成副本；原图存私有 R2 桶，仅后台带 Token 访问

### 7.3 敏感字段加密

- `real_name_enc`：AES-256-GCM，密钥放 KMS / 环境变量
- `password_hash`：bcrypt cost 12（不可逆）
- `totp_secret_enc`：AES-256-GCM
- **一次性密码明文**：仅在 Owner 创建账号 / 重置密码的响应里出现一次，不进日志、不进 Sentry、不写入 `audit_log.payload`
- **密钥轮换**：AES 密钥分版本号（`v1` / `v2`），加密时记录版本，解密自动按版本查表；轮换时先双写再下线旧版

### 7.4 CSRF / XSS

- 写接口要求 `X-CSRF-Token` Header（Cookie + Header 双验证）
- 输入 HTML 转义，富文本走白名单 sanitize
- Response Header：`Content-Security-Policy`、`X-Frame-Options: DENY`、`Strict-Transport-Security`

### 7.5 防截屏 / 防盗图

服务端能做的有限，**客户端水印是主防线**：
- 所有公开图片 CDN URL 带水印副本
- 原图不出 CDN，仅后台带 Token 访问
- 高敏感模特可标记 `watermark_level=high`，水印更明显

## 八、部署与运维

### 8.1 环境

| 环境 | 用途 | 数据 |
| --- | --- | --- |
| dev | 开发联调 | mock |
| staging | 验收 | 脱敏镜像 |
| prod | 生产 | 真实 |

### 8.2 部署架构

```
[Cloudflare DNS + WAF]
        │
        ├── Cloudflare Pages（H5 客户端，主域，邀请制 URL 混淆）
        ├── Cloudflare Pages（Admin Console，admin.* 子域）
        └── Cloudflare Workers（API：公开 + 鉴权）
              ├── Neon Postgres（Serverless，HTTP 驱动）
              ├── Upstash Redis（限流 + Session，HTTPS）
              └── Cloudflare R2（图片 / 视频原始资产 + 缩略 + 水印副本）
```

部署单位：Workers（API）+ Pages（两个前端）= 三个独立 wrangler / pages 项目，同一 Cloudflare account。

### 8.3 监控

- 接口耗时 P95 < 300ms，P99 < 800ms（边缘节点测得）
- 错误率 > 1% 告警（Sentry）
- R2 流量 / Neon Compute Units / Upstash 请求数 / 短信余额告警
- Workers 健康检查：`/health`、`/ready`（边缘节点本地探测，不出公网）

### 8.4 备份

- DB：Neon 自带 PITR（point-in-time recovery）保留 7 天；重要快照人工拉取并归档至独立 R2 桶
- R2：开启 Object Versioning + 7 天保留
- 审计日志：1 年内可查，超期归档至冷 R2 桶

### 10.2 2026-05-28 第二轮锁定（登录与隐私）

| # | 决策项 | 结论 | 影响章节 |
| --- | --- | --- | --- |
| 1 | 登录方式 | ✅ **用户名 + 密码 + 强制 TOTP**；取消手机号、取消短信验证码、取消邮箱 | §2.2 Admin、§3.1、§4.2 |
| 2 | 强制 2FA 范围 | ✅ **所有角色**（Owner / Admin / Operator）一律强制 TOTP | §3.1 |
| 3 | 邀请新管理员 | ✅ **Owner 后台生成一次性密码**，明文仅响应一次，线下当面交付；不通过短信 / 邮件 / IM | §4.7 |
| 4 | 密码重置 | ✅ 同邀请，由 Owner 触发，置 `must_change_password=true` | §4.7 |
| 5 | 2FA 重置 | ✅ Owner 触发，清空 secret 并置 `totp_enrolled=false`，下次登录强制重绑 | §4.7 |
| 6 | 账号失败锁定 | ✅ 连续 5 次密码错误 → 锁定 15 min；Owner 可手动解锁 | §3.1.1、§4.7 |
| 7 | 一次性密码材料 | ✅ 20 字符 URL-safe random；不进日志 / Sentry / 审计 payload | §7.3 |

**决策动机**：业主明确 "完全保证用户隐私不泄漏"。不存手机号 / 邮箱意味着账号系统与任何外部通信通道完全解耦，被拖库时也不会暴露管理员 PII，且系统无第三方短信 / 邮件依赖、月成本进一步压缩。
