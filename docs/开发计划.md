## 〇、本方案是什么

API 契约 / H5 / 部署架构三份文档已经把"长什么样、跑在哪台机器上"答完了。本文档回答"按什么顺序、谁等谁、什么时候算完"。

v2 在 v1 基础上做了一次结构性重排，对应业主 2026-05-29 的**部署形态推翻**决策：

- **栈切换**：CF Workers → VPS Node 22 + Hono on Node；Neon → 本机 Postgres；Upstash REST → ioredis；R2 → 本地磁盘 + sharp；WebCrypto → node:crypto；bcryptjs → bcrypt native
- **既有代码大半要重做**：Phase 0–Step 7（commit `f30c38b` … `69a0f44`）的 25 个 commit 是按 Workers 假设写的（HTTP 驱动、HTTPS REST、`@cf/image-resizing`、Wrangler 部署等），多数需要按新栈重做或切换驱动
- **重排逻辑**：旧 v1 Phase 0 / 1 已经被"切栈"动作整体替代，故 v2 新增 **Phase R（重定基）** 作为起点，把已有 mock 代码迁到新栈，然后从原 Phase 2 起重新推进

## 一、目标与约束

### 1.1 目标

把已冻结的设计落地为可用的生产系统：

| # | 交付物 | 部署形态 |
|---|---|---|
| 1 | H5 客户端 | Caddy 静态托管，主域，邀请制 URL 混淆 |
| 2 | Admin Console | Caddy 静态托管，`admin.*` 子域 + IP 白名单 |
| 3 | API 服务 | systemd 托管的 Node + Hono on Node，监听 `127.0.0.1:3000` |
| 4 | 数据存储 | VPS 本机 PostgreSQL 16 + Redis 7 + 本地磁盘 `/var/chiyan/media` |
| 5 | 反代 / TLS / 静态 | Caddy 2 + Cloudflare Origin CA 证书 |
| 6 | DNS / CDN / WAF | Cloudflare proxied 模式 |
| 7 | 备份 | `pg_dump` + `rsync media/` → Backblaze B2，每日 cron |
| 8 | 监控 | Sentry + Caddy access log + systemd journal |

### 1.2 硬约束（不可妥协）

- **海外部署 / 不备案 / 不接审核**（业主 2026-05-28）
- **邀请制 = URL 混淆**：不做登录 / 邀请码，靠不可猜 URL + noindex + 空 robots.txt
- **不使用 Docker**（业主 2026-05-29）：全部 systemd + apt 原生包
- **2 GB RAM 是硬上限**：Postgres + Redis + Node + sharp + Caddy 合计 ≤ 1.7G，详见 [API 文档 §一.4](./后台管理系统接口设计方案.md)
- **未成年模特强制隐去身体数据**：覆盖整体公开策略
- **真名 / 联系方式 / 证件**：永不公开
- **一次性密码 / TOTP secret**：不进日志 / Sentry / 审计 payload
- **大陆 best-effort 可达**：不签 SLA，预留 China-friendly CDN 切换通道

### 1.3 非目标（明确不做）

- ❌ 客户端登录 / 收藏 / 个性化
- ❌ 客户端在线预约表单（QQ 全程替代）
- ❌ PDF 模卡导出
- ❌ 高级搜索 / 推荐算法 / 虚拟滚动等大数据量优化
- ❌ 微信全家桶（业主 2026-05-29 再次确认）
- ❌ ICP 备案 / 内容审核接入
- ❌ Docker / k8s / Grafana / Prometheus / ELK
- ❌ Cloudflare Workers / Pages / R2（业主 2026-05-29 推翻）
- ❌ 手机端后台 / 后台 PWA（后置 P3）

### 2.2 关键栈切换（v2 对 v1 的差量）

| 维度 | v1（CF Workers） | v2（VPS Node） |
| --- | --- | --- |
| 运行时 | `wrangler dev` / Workers Isolate | `node --watch` / Hono on Node |
| Hono adapter | 默认 `app.fetch` 直接被 Workers 拉起 | `serve({ fetch: app.fetch, port: 3000 })`（`@hono/node-server`） |
| DB 驱动 | `@neondatabase/serverless` HTTP | `pg` Pool TCP + `drizzle-orm/node-postgres` |
| Redis | `@upstash/redis` HTTPS REST | `ioredis` TCP 长连 |
| 鉴权 | `hono/jwt` HS256 + bcryptjs + WebCrypto | `jose` ES256 + bcrypt native + node:crypto |
| 上传 | `/admin/media/sign` + R2 presigned PUT + `/register` | 单步 `POST /admin/media/upload` multipart + sharp 同步处理 |
| 图片处理 | `@cf/image-resizing` | sharp（`p-limit(2)` 并发） |
| 部署 | `wrangler deploy` | GitHub Actions → SSH → `git pull && pnpm build && systemctl restart chiyan-api` |

## 三、技术栈复述

完整选型见 [API 文档 §一.3](./后台管理系统接口设计方案.md)。开发期特别注意：

| 维度 | 实际约束 |
|---|---|
| 运行时 | Node 22 LTS；单进程，无 cluster；`--max-old-space-size=384` |
| Hono | `@hono/node-server` adapter；可自由用 `fs / net / child_process / pg / ioredis / sharp` |
| 数据库 | `pg.Pool` 最大 10 连接；Drizzle `node-postgres` driver；连接 `127.0.0.1:5432` |
| Redis | `ioredis` 单实例长连；`127.0.0.1:6379`；用于限流 + jti 黑名单 + challenge token |
| 鉴权 | 自建 JWT（jose ES256）+ HttpOnly + Secure Cookie + 强制 TOTP + jti Redis 黑名单 |
| 密码 | bcrypt native cost 12（2C 机器一次登录 ~80ms） |
| 加密 | node:crypto AES-256-GCM；密钥从 systemd EnvironmentFile 读 |
| 上传 | Hono multipart；sharp 同步生成 cover/gallery/thumbs/originals，`p-limit(2)` 限并发 |
| TOTP | otpauth（不变） |

### Phase R — 重定基（约 4 工日）

**前置**：业主 2026-05-29 决策推翻 CF 全栈 Serverless（业主原话："实在是太麻烦了"），现有 `apps/api` 内的 Workers mock 代码需要全面切换驱动。

**任务**

1. **`apps/api` 入口换 adapter**：
   - 移除 `wrangler.toml` / `wrangler.json`，新增 `apps/api/src/server.ts` 用 `@hono/node-server` 的 `serve({ fetch: app.fetch, port: 3000 })`
   - 删 `nodejs_compat` 等 Workers 兼容 flag 路径
2. **数据库驱动切换**：
   - `packages/db` 把 `drizzle-orm/neon-http` 切到 `drizzle-orm/node-postgres`，导出 `db = drizzle(new Pool({ connectionString }))`
   - 现有 schema + migration 复用，**无需重写 schema**
   - 本机起 Postgres，跑 `pnpm --filter db migrate:deploy` 验证 schema 一致
3. **Redis 切换**：
   - `apps/api` 内 `@upstash/redis` 全部替换为 `ioredis`；wrapper 接口签名保持不变，调用点零改动
   - 限流中间件、jti 黑名单、challenge token 三个用法跑通本机集成测试
4. **密码 / 加密 / JWT 切换**：
   - `bcryptjs` → `bcrypt`（native）；接口一致，性能预算从 ~600ms 降到 ~80ms
   - WebCrypto `crypto.subtle` → `node:crypto`；AES-256-GCM 工具签名保持不变
   - `hono/jwt` HS256 → `jose` ES256；密钥从 `EnvironmentFile` 读
5. **媒体上传重构**：
   - 删 `/admin/media/sign` + `/admin/media/register` 两端点
   - 新增 `POST /admin/media/upload`（multipart）：Hono 接收 → magic number 校验 → sharp 同步生成多尺寸 + 水印 + WebP → 落盘 `/var/chiyan/media/models/<id>/...` → 写 MediaAsset 表 → 返回 `asset_id` + `cdn_url`
   - 引入 `p-limit(2)` 控制 sharp 并发
6. **本机端到端 smoke**：
   - `docker-compose`？❌ 不用。直接 `pnpm dev:api` + 本机 Postgres/Redis
   - 跑全套现有集成测试（25 commits 累积下来约 24 个 test suite），90% 以上应该绿；剩下涉及 Workers / R2 / Upstash REST 假设的逐个改

**退出标准**

- `pnpm dev` 在三个 app 都能起本地
- `pnpm typecheck` / `pnpm test` 全 green
- `apps/api` 监听 `:3000`，本机 `curl localhost:3000/health` 返回 200
- 端到端跑通：本机起 API → Postman 完成"新增账号 → 首登 → 改密 → 绑 TOTP → 创建模特 → 上传图片"全流程
- `wrangler` / `@neondatabase/serverless` / `@upstash/redis` / `@cf/image-resizing` / `bcryptjs` / WebCrypto 全部从 `package.json` 移除，`grep -rE "wrangler|neon-http|upstash|cf/image-resizing" apps packages` 0 命中

**风险**

- sharp 在 macOS / Linux 跨平台原生依赖：CI 用 Linux x64 即可避免；M 系列 mac 本地装走 npm 通道
- bcrypt native：本地 mac 装 prebuild 二进制，CI 同样
- 测试快照 / fixtures 可能引用 R2 URL 格式；批量改

## 七、风险与备选

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| Phase R 切栈遗留 bug 拖周期 | 高 | 中 | 单独 Phase 独立绿测；不与新功能交叉 |
| 2G RAM 高峰期吃紧（图片处理 + 慢查） | 中 | 高 | sharp `p-limit(2)`；Postgres `shared_buffers=256MB`；systemd `MemoryMax` 兜底；告警 1.7G 提前介入 |
| 单台 VPS 主机商挂 | 低 | 高 | 备份脚本日推 B2；预案：新开 VPS → §二初始化 → 恢复 → DNS 切，RTO ≤ 4h |
| 大陆访问 best-effort 不达预期 | 中 | 中 | 预留 China-friendly CDN 切换；DNS 切换无业务变更 |
| Admin 丢失 TOTP 且 Owner 双盲 | 低 | 高 | Owner 首绑时离线保存 secret 二维码到保险柜 |
| 一次性密码线下交付泄漏 | 低 | 中 | 强制首登改密 + 强制 TOTP；密码 history 5 条 |
| Cloudflare Origin CA 证书私钥泄漏 | 低 | 中 | 1Password 单独条目；如泄漏立刻 CF 后台 revoke 重签 |
| sharp 在 VPS 上水印效果不理想 | 低 | 低 | 备选：SVG overlay；本地预览验证后再上 prod |
| 业主品牌 VI 拖延 | 高 | 低 | token 化，上线前任意时间替换 |
| 设计 / 开发完全串行导致延期 | 中 | 高 | 严格章节级冻结；开发可先用占位 UI 跑通数据流 |
| 单人全栈 burnout / 病假 | 中 | 高 | 文档先行（API 契约 + schema + 部署架构都锁定），任何一段都可被外部接手 |
